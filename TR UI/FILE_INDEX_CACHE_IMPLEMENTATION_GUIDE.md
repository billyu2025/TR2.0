## 1. 数据库表设计

### 1.1 主索引表：`file_index_cache`

**表结构设计：**

```
字段说明：
- id: INTEGER PRIMARY KEY AUTOINCREMENT
  → 主键，自增ID

- file_path: TEXT NOT NULL UNIQUE
  → 文件的完整绝对路径
  → 唯一索引，防止重复

- file_name: TEXT NOT NULL
  → 文件名（不含路径）
  → 用于快速显示

- folder_path: TEXT NOT NULL
  → 文件所属的文件夹路径
  → 用于按文件夹类型查询

- folder_type: TEXT NOT NULL
  → 文件夹类型枚举值：
    * 'Stockist Cert'
    * 'Private Formal'
    * 'Private Prelim'
    * 'IAT Formal'
    * 'IAT Prelim'
  → 用于快速过滤特定类型的文件夹

- file_size: INTEGER
  → 文件大小（字节）
  → 用于验证文件是否被修改

- modified_time: REAL
  → 文件最后修改时间（Unix时间戳）
  → 用于检测文件变化

- created_time: TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
  → 索引记录创建时间

- last_checked: TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
  → 最后检查时间（用于增量更新）

- extracted_keywords: TEXT
  → 从文件名中提取的关键词（JSON格式）
  → 例如：["12345", "ABC123", "STOCKIST001"]
  → 用于快速匹配查询

- file_hash: TEXT
  → 文件内容的哈希值（可选，用于检测文件内容变化）
  → 可以只计算文件名的哈希，不计算整个文件
```

**索引设计：**

```
1. 主键索引：id（自动创建）

2. 唯一索引：file_path
   → 确保每个文件只有一条记录

3. 复合索引：folder_type + file_name
   → 用于按文件夹类型和文件名查询

4. 索引：extracted_keywords（如果SQLite支持JSON索引）
   → 或者使用全文搜索（FTS5）

5. 索引：last_checked
   → 用于增量更新时快速查找需要检查的文件
```

### 1.2 辅助表：`file_index_metadata`

**用途：** 存储索引的元数据信息

```
字段：
- key: TEXT PRIMARY KEY
  → 配置键名

- value: TEXT
  → 配置值

- updated_at: TEXT DEFAULT CURRENT_TIMESTAMP
  → 更新时间

存储的信息：
- 'last_full_scan': 最后一次全量扫描的时间
- 'total_files_indexed': 索引的文件总数
- 'index_version': 索引版本号（用于兼容性检查）
- 'scan_status': 扫描状态（'idle', 'scanning', 'updating'）
```

## 2. 索引建立机制

### 2.1 全量索引建立流程

**触发时机：**
- 系统首次启动
- 手动触发重建索引
- 索引损坏或版本不兼容

**建立步骤：**

#### 步骤1：准备工作
1. **检查数据库连接**
   - 连接到 `data_3years.db`
   - 启用 WAL 模式（提高并发性能）
   - 设置超时时间（30秒）

2. **创建表结构**
   - 检查 `file_index_cache` 表是否存在
   - 如果不存在，创建表并建立索引
   - 如果存在但结构不匹配，备份旧表并重建

3. **获取基础路径**
   - 从环境变量读取 `STOCKIST_TEST_FOLDER`
   - 验证路径是否存在
   - 记录到元数据表

#### 步骤2：扫描文件系统
1. **定义扫描范围**
   ```
   需要扫描的文件夹：
   - {base_folder}/Stockist Cert
   - {base_folder}/Private Formal
   - {base_folder}/Private Prelim
   - {base_folder}/IAT Formal
   - {base_folder}/IAT Prelim
   ```

2. **遍历策略**
   - 使用 `os.walk()` 递归遍历每个文件夹
   - 只处理 `.pdf` 文件（忽略其他格式）
   - 记录遍历进度（每1000个文件输出一次）

3. **提取文件信息**
   对每个PDF文件：
   - 获取完整路径（绝对路径）
   - 获取文件名
   - 获取文件大小
   - 获取修改时间（转换为Unix时间戳）
   - 确定文件夹类型（根据路径判断）
   - 提取关键词（见下方说明）

#### 步骤3：关键词提取策略

**目标：** 从文件名中提取可能用于匹配的关键词

**提取规则：**
1. **数字序列提取**
   - 提取所有连续的数字序列（长度>=3）
   - 例如：`"ABC12345DEF"` → `["12345"]`
   - 例如：`"STOCKIST-001-2023"` → `["001", "2023"]`

2. **字母数字组合**
   - 提取字母+数字的组合（如证书编号格式）
   - 例如：`"STOCKIST001"` → `["STOCKIST001"]`
   - 例如：`"RM-DN-12345"` → `["RM", "DN", "12345"]`

3. **分隔符处理**
   - 按常见分隔符分割：`-`, `_`, ` `, `.`
   - 保留有意义的片段（长度>=2）

4. **去重和规范化**
   - 转换为小写统一比较
   - 去除空字符串
   - 去除重复项

**示例：**
```
文件名：`STOCKIST_CERT_12345_RM_DN_67890.pdf`
提取结果：["stockist", "cert", "12345", "rm", "dn", "67890", "stockistcert", "rmdn"]
存储为JSON：["12345", "67890", "STOCKIST_CERT_12345", "RM_DN_67890"]
```

#### 步骤4：批量写入数据库
1. **批量插入策略**
   - 使用事务批量插入（每批1000条）
   - 使用 `INSERT OR REPLACE` 处理重复
   - 记录插入进度

2. **错误处理**
   - 文件访问权限错误：跳过并记录
   - 数据库写入错误：回滚当前批次，记录错误
   - 继续处理其他文件

3. **更新元数据**
   - 更新 `last_full_scan` 时间
   - 更新 `total_files_indexed` 计数
   - 设置 `scan_status` 为 'idle'

### 2.2 增量索引建立

**触发时机：**
- 定期自动检查（如每小时）
- 文件下载操作前检查
- 手动触发

**建立步骤：**

1. **获取需要检查的文件**
   - 查询 `last_checked` 超过阈值的记录（如1小时前）
   - 或者检查文件夹中是否有新文件

2. **对比文件系统**
   - 遍历文件夹，获取当前文件列表
   - 与数据库中的记录对比
   - 识别：新增文件、删除文件、修改文件

3. **更新数据库**
   - 新增：插入新记录
   - 删除：标记为已删除（或直接删除记录）
   - 修改：更新文件大小、修改时间、重新提取关键词

## 3. 索引更新机制

### 3.1 更新策略选择

**策略1：定时全量扫描**
- **频率：** 每天一次（如凌晨2点）
- **优点：** 保证数据完整性
- **缺点：** 资源消耗大
- **适用：** 文件变化频繁的场景

**策略2：定时增量更新**
- **频率：** 每小时一次
- **优点：** 资源消耗小
- **缺点：** 可能遗漏快速变化
- **适用：** 文件变化不频繁的场景（推荐）

**策略3：事件驱动更新**
- **触发：** 文件系统监控（如 watchdog）
- **优点：** 实时性最好
- **缺点：** 实现复杂，需要额外依赖
- **适用：** 对实时性要求极高的场景

**推荐方案：** 策略2（定时增量更新）+ 策略1（每天全量扫描一次）

### 3.2 增量更新流程

#### 步骤1：确定更新范围
1. **时间窗口检查**
   - 查询 `last_checked < (当前时间 - 1小时)` 的记录
   - 限制每次更新的文件数量（如10000个）

2. **文件夹检查**
   - 检查每个文件夹的最后修改时间
   - 如果文件夹未修改，跳过该文件夹下的所有文件

#### 步骤2：文件对比
1. **获取数据库中的文件列表**
   ```sql
   SELECT file_path, modified_time, file_size 
   FROM file_index_cache 
   WHERE folder_type = ? AND last_checked < ?
   ```

2. **获取文件系统中的文件列表**
   - 遍历文件夹，获取当前所有PDF文件
   - 构建 `{file_path: (size, mtime)}` 字典

3. **对比差异**
   - **新增文件：** 文件系统中存在，数据库中不存在
   - **删除文件：** 数据库中存在，文件系统中不存在
   - **修改文件：** 文件大小或修改时间不同

#### 步骤3：更新数据库
1. **处理新增文件**
   - 提取文件信息
   - 提取关键词
   - 插入新记录

2. **处理删除文件**
   - 标记为已删除（添加 `is_deleted` 标志）
   - 或直接删除记录（如果确定文件不会再出现）

3. **处理修改文件**
   - 更新 `file_size`、`modified_time`
   - 重新提取关键词（文件名可能变化）
   - 更新 `last_checked` 时间

4. **更新所有文件的 `last_checked`**
   - 即使文件未变化，也更新检查时间

### 3.3 更新时机设计

**自动更新：**
- 系统启动时检查（如果距离上次更新超过1小时）
- 后台定时任务（每小时执行一次）
- 文件下载操作前（如果距离上次更新超过30分钟）

**手动更新：**
- 管理员手动触发全量重建
- API接口触发增量更新

## 4. 查询优化方案

### 4.1 查询流程改造

**原流程（慢）：**
```
1. 接收查询请求（关键词列表）
2. 遍历文件夹（os.walk）
3. 检查每个文件名是否包含关键词
4. 返回匹配的文件路径
```

**新流程（快）：**
```
1. 接收查询请求（关键词列表、文件夹类型）
2. 在数据库中查询匹配的文件
3. 验证文件是否存在（可选，快速检查）
4. 返回匹配的文件路径
```

### 4.2 数据库查询策略

#### 策略1：关键词匹配查询

**查询逻辑：**
```sql
-- 伪SQL示例
SELECT file_path, file_name, folder_type
FROM file_index_cache
WHERE folder_type IN (?, ?, ?)  -- 指定文件夹类型
  AND (
    -- 方式1：使用LIKE（简单但可能慢）
    file_name LIKE '%keyword1%' OR
    file_name LIKE '%keyword2%' OR
    ...
    
    -- 方式2：使用JSON函数（如果SQLite版本支持）
    json_extract(extracted_keywords, '$') LIKE '%keyword1%'
    
    -- 方式3：使用全文搜索FTS5（最快，需要额外配置）
    file_name MATCH 'keyword1 OR keyword2'
  )
  AND is_deleted = 0  -- 排除已删除的文件
```

**优化建议：**
- 如果关键词数量少（<5个），使用多个 `LIKE` 条件
- 如果关键词数量多，考虑使用全文搜索（FTS5）
- 添加 `folder_type` 过滤，减少扫描范围

#### 策略2：关键词提取匹配

**思路：** 在插入时提取关键词，查询时直接匹配关键词

**实现：**
1. 查询时，将输入的关键词规范化（转小写、去空格）
2. 在 `extracted_keywords` JSON字段中查找
3. 使用SQLite的JSON函数或字符串匹配

#### 策略3：缓存热门查询

**思路：** 对常见查询结果进行缓存

**实现：**
1. 使用内存缓存（如Python的 `functools.lru_cache`）
2. 缓存键：`(folder_type, tuple(sorted(keywords)))`
3. 缓存时间：5-10分钟
4. 缓存大小：限制最多1000个查询结果

### 4.3 查询接口设计

**接口1：按关键词查询**
```
输入：
- keywords: List[str]  # 关键词列表
- folder_types: List[str]  # 文件夹类型列表（可选）
- limit: int  # 返回结果数量限制（可选）

输出：
- file_paths: List[str]  # 匹配的文件路径列表
- total_count: int  # 总匹配数
- query_time: float  # 查询耗时（毫秒）
```

**接口2：按订单号查询（高级）**
```
输入：
- order_no: int
- folder_types: List[str]

处理：
1. 从数据库获取订单的 stockist_cert 和 rm_dn_no
2. 将这些值作为关键词查询文件索引
3. 返回匹配的文件

输出：
- file_paths: List[str]
- matched_keywords: Dict[str, List[str]]  # 每个关键词匹配到的文件
```

### 4.4 性能优化技巧

1. **索引优化**
   - 确保 `folder_type` 有索引
   - 确保 `file_path` 有唯一索引
   - 考虑为 `extracted_keywords` 创建索引

2. **查询优化**
   - 先按 `folder_type` 过滤，减少扫描范围
   - 使用 `LIMIT` 限制返回结果
   - 避免使用 `SELECT *`，只选择需要的字段

3. **结果验证**
   - 查询后快速验证文件是否存在（`os.path.exists`）
   - 如果文件不存在，从索引中删除该记录
   - 但不要阻塞查询，可以异步清理

## 5. 维护机制

### 5.1 定期维护任务

#### 任务1：清理无效记录
**频率：** 每天一次

**操作：**
1. 查询所有索引记录
2. 检查文件是否仍然存在
3. 删除不存在的文件记录
4. 记录清理统计信息

#### 任务2：重建索引
**频率：** 每周一次（可选）

**操作：**
1. 备份当前索引表
2. 清空索引表
3. 执行全量扫描
4. 对比新旧索引，记录差异

#### 任务3：优化数据库
**频率：** 每月一次

**操作：**
1. 执行 `VACUUM` 清理数据库碎片
2. 重建索引（`REINDEX`）
3. 分析表统计信息（`ANALYZE`）

### 5.2 监控和告警

#### 监控指标
1. **索引完整性**
   - 索引文件数 vs 实际文件数
   - 差异超过10%时告警

2. **查询性能**
   - 平均查询时间
   - 超过1秒的查询记录日志

3. **更新延迟**
   - 最后更新时间
   - 超过24小时未更新时告警

4. **存储空间**
   - 索引表大小
   - 超过100MB时考虑清理

#### 告警机制
- 记录到日志文件
- 发送邮件通知（可选）
- 在管理界面显示警告

### 5.3 故障恢复

#### 场景1：索引表损坏
**处理：**
1. 检测到表损坏（SQLite错误）
2. 备份损坏的表
3. 删除并重建表
4. 触发全量索引重建

#### 场景2：文件系统变化
**处理：**
1. 检测到大量文件变化（超过50%）
2. 记录警告日志
3. 触发全量索引重建
4. 通知管理员

#### 场景3：性能下降
**处理：**
1. 监控查询时间
2. 如果平均查询时间超过阈值
3. 分析慢查询日志
4. 优化索引或查询语句
5. 考虑重建索引

### 5.4 管理接口

**接口1：获取索引状态**
```
GET /api/file-index/status

返回：

  "total_files": 12345,
  "last_full_scan": "2024-01-15 02:00:00",

```

```
返回：
{
  "success": true,
  "message": "索引重建已启动",
  "estimated_time_minutes": 30

```

**接口3：手动触发增量更新**
```
POST /api/file-index/update

返回：
{
  "success": true,
  "files_added": 10,
  "files_updated": 5,

```

```
  "success": true,
```

## 7. 注意事项

### 7.1 性能考虑

1. **首次索引建立**
   - 可能需要较长时间（取决于文件数量）
   - 建议在系统空闲时执行
   - 显示进度，允许中断和恢复

2. **数据库大小**
   - 索引表可能占用一定空间
   - 定期清理无效记录
   - 考虑压缩存储

3. **并发访问**
   - 使用WAL模式支持并发读取
   - 更新时使用事务避免冲突
   - 考虑读写锁机制

### 7.2 数据一致性

1. **文件变化检测**
   - 文件可能在索引更新时被修改
   - 使用文件大小+修改时间双重验证
   - 定期全量扫描保证一致性

2. **错误处理**
   - 文件访问错误不应影响整体索引
   - 记录错误日志便于排查
   - 提供手动修复机制

### 7.3 兼容性

1. **向后兼容**
   - 保留原有的文件查找方法作为降级方案
   - 如果索引不可用，自动回退
   - 平滑迁移，不影响现有功能

2. **版本管理**
   - 索引表结构版本化
   - 支持结构升级
   - 提供迁移脚本
