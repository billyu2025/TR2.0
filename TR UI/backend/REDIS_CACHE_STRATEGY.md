# Redis 緩存策略建議

## 系統數據分析

### 高頻訪問的 API 端點

根據系統分析，以下端點訪問頻率較高：

1. **訂單列表查詢** - `/api/orders/list` (GET)
2. **材料搜索** - `/api/materials/search/<tag_no>` (GET)
3. **用戶信息** - `/api/auth/me` (GET)
4. **用戶列表** - `/api/admin/users` (GET)
5. **文件索引狀態** - `/api/file-index/status` (GET)
6. **按 Job No 分組** - `/api/orders/group-by-job-no` (GET)

## 推薦緩存的數據（按優先級）

### 優先級 1：高頻訪問 + 相對穩定（強烈推薦）

#### 1. 訂單列表查詢結果

**端點：** `/api/orders/list`

**為什麼適合緩存：**
- ✅ **訪問頻率極高**：用戶經常查詢訂單列表
- ✅ **查詢成本高**：涉及 JOIN、排序、分頁
- ✅ **數據相對穩定**：訂單數據不會頻繁變化
- ✅ **多用戶共享**：多個用戶可能查詢相同條件

**緩存策略：**
```python
# 緩存鍵格式
cache_key = f"orders:list:{page}:{per_page}:{order_no}:{job_no}:{dn_no}:{start_date}:{end_date}:{tab}"

# 緩存時間
cache_ttl = 300  # 5分鐘（訂單數據相對穩定）
```

**緩存收益：**
- 減少數據庫查詢壓力
- 提升響應速度（從 500ms → 10ms）
- 支持高並發訪問

**注意事項：**
- 訂單更新時需要清除相關緩存
- 分頁結果需要分別緩存

#### 2. 材料搜索結果

**端點：** `/api/materials/search/<tag_no>`

**為什麼適合緩存：**
- ✅ **訪問頻率高**：用戶頻繁搜索材料
- ✅ **查詢成本中等**：需要查詢 materials_com 表
- ✅ **數據穩定**：材料數據很少變化
- ✅ **結果可重用**：相同 Tag No 的查詢結果相同

**緩存策略：**
```python
# 緩存鍵格式
cache_key = f"materials:search:{tag_no}"

# 緩存時間
cache_ttl = 3600  # 1小時（材料數據非常穩定）
```

**緩存收益：**
- 減少重複查詢
- 提升搜索響應速度
- 降低數據庫負載

#### 3. 用戶信息（會話數據）

**端點：** `/api/auth/me`

**為什麼適合緩存：**
- ✅ **訪問頻率極高**：每次頁面加載都可能查詢
- ✅ **查詢成本低但頻繁**：需要查詢用戶表
- ✅ **數據相對穩定**：用戶信息不會頻繁變化
- ✅ **會話相關**：可以與 Session 結合

**緩存策略：**
```python
# 緩存鍵格式
cache_key = f"user:profile:{user_id}"

# 緩存時間
cache_ttl = 1800  # 30分鐘（與 Session 時間對應）
```

**緩存收益：**
- 減少數據庫查詢
- 提升用戶體驗
- 支持會話管理

**注意事項：**
- 用戶信息更新時需要清除緩存
- 可以與 Session 緩存結合

### 優先級 2：中頻訪問 + 計算成本高（推薦）

#### 4. 按 Job No 分組結果

**端點：** `/api/orders/group-by-job-no`

**為什麼適合緩存：**
- ✅ **計算成本高**：需要 GROUP BY 和聚合
- ✅ **數據相對穩定**：分組結果不會頻繁變化
- ✅ **結果可重用**：相同查詢條件結果相同

**緩存策略：**
```python
# 緩存鍵格式
cache_key = f"orders:group-by-job:{filters_hash}"

# 緩存時間
cache_ttl = 600  # 10分鐘
```

#### 5. 文件索引狀態

**端點：** `/api/file-index/status`

**為什麼適合緩存：**
- ✅ **查詢成本中等**：需要統計文件數量
- ✅ **數據變化不頻繁**：索引狀態不會頻繁變化
- ✅ **管理功能**：管理員查看，頻率中等

**緩存策略：**
```python
# 緩存鍵格式
cache_key = "file_index:status"

# 緩存時間
cache_ttl = 300  # 5分鐘
```

**注意事項：**
- 文件索引更新時需要清除緩存

#### 6. 用戶列表（管理員功能）

**端點：** `/api/admin/users`

**為什麼適合緩存：**
- ✅ **訪問頻率中等**：管理員查看用戶列表
- ✅ **數據相對穩定**：用戶列表不會頻繁變化
- ✅ **查詢成本中等**：需要查詢用戶表

**緩存策略：**
```python
# 緩存鍵格式
cache_key = "admin:users:list"

# 緩存時間
cache_ttl = 300  # 5分鐘
```

**注意事項：**
- 用戶創建/更新/刪除時需要清除緩存

### 優先級 3：會話和臨時數據（可選）

#### 7. Session 數據

**為什麼適合緩存：**
- ✅ **訪問頻率極高**：每次請求都需要驗證
- ✅ **臨時數據**：會話數據本身就是臨時的
- ✅ **Redis 天然適合**：Redis 常用於 Session 存儲

**緩存策略：**
```python
# 緩存鍵格式
cache_key = f"session:{session_id}"

# 緩存時間
cache_ttl = 86400  # 24小時（與 Session TTL 對應）
```

#### 8. 用戶權限緩存

**為什麼適合緩存：**
- ✅ **訪問頻率高**：每次請求都需要檢查權限
- ✅ **數據相對穩定**：用戶權限不會頻繁變化
- ✅ **查詢成本中等**：需要查詢用戶表和權限表

**緩存策略：**
```python
# 緩存鍵格式
cache_key = f"user:permissions:{user_id}"

# 緩存時間
cache_ttl = 1800  # 30分鐘
```

## 不推薦緩存的數據

### 1. PDF 生成結果

**端點：** `/api/pdf/generate`

**為什麼不適合：**
- ❌ **動態生成**：每次生成可能不同
- ❌ **文件大**：PDF 文件可能很大，不適合內存
- ❌ **個性化**：每個用戶的 PDF 可能不同
- ❌ **已存儲**：生成的 PDF 已存儲在文件系統

**替代方案：**
- 使用文件系統存儲生成的 PDF
- 緩存 PDF 狀態（是否已生成）

### 2. 文件下載

**端點：** `/api/pdf/download`, `/api/stockist-test/download-by-order`

**為什麼不適合：**
- ❌ **文件大**：下載的文件可能很大（幾 MB 到幾 GB）
- ❌ **動態生成**：ZIP 文件是動態生成的
- ❌ **內存限制**：Redis 內存有限，不適合存儲大文件

**替代方案：**
- 使用文件系統存儲
- 緩存下載任務狀態

### 3. 實時數據

**為什麼不適合：**
- ❌ **數據變化頻繁**：需要實時性
- ❌ **緩存失效頻繁**：緩存可能立即失效
- ❌ **緩存收益低**：緩存意義不大

**示例：**
- 下載任務進度
- 實時統計數據

### 4. 寫入操作結果

**端點：** `/api/tr-fill-in/save`, `/api/tr-fill-in/update`

**為什麼不適合：**
- ❌ **寫入操作**：不是查詢操作
- ❌ **需要立即生效**：寫入後需要立即反映
- ❌ **緩存意義不大**：不是讀取操作

**替代方案：**
- 寫入後清除相關緩存
- 使用緩存失效策略

## 緩存策略建議

### 1. 緩存鍵命名規範

```
{模組}:{操作}:{參數1}:{參數2}:...
```

**示例：**
```
orders:list:1:100:129601::2024-01-01:2024-12-31:stocklist-test
materials:search:410340
user:profile:123
admin:users:list
```

### 2. 緩存時間（TTL）建議

| 數據類型 | TTL | 原因 |
|---------|-----|------|
| **訂單列表** | 5分鐘 | 數據相對穩定，但可能更新 |
| **材料搜索** | 1小時 | 數據非常穩定 |
| **用戶信息** | 30分鐘 | 與 Session 對應 |
| **用戶列表** | 5分鐘 | 管理功能，更新不頻繁 |
| **文件索引狀態** | 5分鐘 | 狀態變化不頻繁 |
| **Session** | 24小時 | 與 Session TTL 對應 |

### 3. 緩存失效策略

#### 主動失效（推薦）

**寫入操作時清除相關緩存：**

```python
# 訂單更新時
def update_order(order_no):
    # 更新數據庫
    # ...
    # 清除相關緩存
    cache.delete(f"orders:list:*")  # 清除所有訂單列表緩存
    cache.delete(f"orders:detail:{order_no}")  # 清除特定訂單緩存
```

#### 被動失效（TTL）

**設置合理的 TTL，讓緩存自動過期：**

```python
# 設置緩存，5分鐘後自動過期
cache.set("orders:list:...", data, ttl=300)
```

### 4. 緩存命中率優化

#### 使用參數哈希

**問題：** 查詢參數多，緩存鍵可能很長

**解決：** 使用參數的哈希值

```python
import hashlib

# 生成參數哈希
params = {"page": 1, "per_page": 100, "order_no": "129601"}
params_str = json.dumps(params, sort_keys=True)
params_hash = hashlib.md5(params_str.encode()).hexdigest()

cache_key = f"orders:list:{params_hash}"
```

#### 分層緩存

**熱點數據：** 使用更長的 TTL
**普通數據：** 使用標準 TTL

## 預期效果

### 性能提升

| 端點 | 當前響應時間 | 緩存後響應時間 | 提升 |
|------|------------|--------------|------|
| **訂單列表** | 500ms | 10ms | **50倍** |
| **材料搜索** | 100ms | 5ms | **20倍** |
| **用戶信息** | 50ms | 2ms | **25倍** |

### 數據庫負載降低

- **訂單列表查詢**：減少 80-90% 的數據庫查詢
- **材料搜索**：減少 95% 的數據庫查詢
- **整體負載**：降低 60-70%

### 並發能力提升

- 支持更多並發用戶
- 減少數據庫連接數
- 提升系統穩定性

## 注意事項

### 1. 緩存一致性

- 寫入操作時需要清除相關緩存
- 使用緩存版本號或時間戳
- 考慮使用緩存失效策略

### 2. 內存管理

- 設置 Redis 內存限制
- 使用 LRU 淘汰策略
- 監控緩存使用情況

### 3. 錯誤處理

- Redis 連接失敗時降級到數據庫查詢
- 緩存錯誤不應該影響系統運行
- 記錄緩存命中率

### 4. 監控和調試

- 監控緩存命中率
- 監控緩存大小
- 記錄緩存操作日誌

1. ✅ **訂單列表查詢** - 收益最大
2. ✅ **材料搜索結果** - 簡單有效
3. ✅ **用戶信息** - 高頻訪問

### 推薦緩存

4. ✅ **按 Job No 分組** - 計算成本高
5. ✅ **文件索引狀態** - 管理功能
6. ✅ **用戶列表** - 管理功能

### 不推薦緩存

- ❌ PDF 生成和下載（文件大）
- ❌ 實時數據（變化頻繁）
- ❌ 寫入操作（不是查詢）

**建議：** 先實施第一階段的 3 個緩存，這些收益最大且實施簡單。根據實際效果再決定是否實施其他緩存。
