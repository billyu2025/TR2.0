# 全局錯誤處理器說明

## 什麼是全局錯誤處理器？

全局錯誤處理器是一個**統一捕獲和處理所有未預期錯誤**的機制。它就像一個"安全網"，確保即使代碼中沒有明確處理的錯誤，也不會導致整個系統崩潰。

## 類比說明

### 沒有全局錯誤處理器

想像一個餐廳：

```
顧客點菜 → 服務員處理
  ↓
如果遇到問題（比如菜單上沒有這道菜）
  ↓
服務員不知道怎麼處理
  ↓
直接崩潰，整個餐廳停止服務 ❌
```

### 有全局錯誤處理器

```
顧客點菜 → 服務員處理
  ↓
如果遇到問題
  ↓
全局錯誤處理器（經理）介入
  ↓
統一處理：道歉、提供替代方案、記錄問題
  ↓
餐廳繼續正常運營 ✅
```

## 為什麼需要全局錯誤處理器？

### 問題1：未捕獲的錯誤導致服務崩潰

**沒有全局錯誤處理器：**
```python
@app.route('/api/orders/list')
def get_orders_list():
    # 如果這裡發生未預期的錯誤
    result = some_function()  # 假設這裡出錯了
    return jsonify(result)
    # 錯誤沒有被 try/except 捕獲
    # → Flask 返回 500 錯誤
    # → 用戶看到不友好的錯誤信息
    # → 可能導致服務不穩定
```

**有全局錯誤處理器：**
```python
# 全局錯誤處理器自動捕獲所有未處理的錯誤
@app.errorhandler(Exception)
def handle_exception(e):
    # 統一處理錯誤
    # 記錄詳細錯誤信息
    # 返回友好的錯誤響應
    # 系統繼續運行
```

### 問題2：錯誤響應不統一

**沒有全局錯誤處理器：**
- 不同端點返回的錯誤格式不同
- 有些返回 JSON，有些返回 HTML
- 前端難以統一處理

**有全局錯誤處理器：**
- 所有錯誤都返回統一的 JSON 格式
- 前端可以統一處理錯誤
- 用戶體驗更好

### 問題3：錯誤信息洩漏

**沒有全局錯誤處理器：**
- 可能返回詳細的錯誤堆棧
- 可能洩漏敏感信息（文件路徑、數據庫結構等）
- 安全風險

**有全局錯誤處理器：**
- 生產環境返回通用錯誤信息
- 詳細錯誤僅記錄在日誌中
- 保護系統安全

## 全局錯誤處理器如何工作？

### 工作流程

```
用戶請求 → API 端點處理
  ↓
發生錯誤（未捕獲）
  ↓
全局錯誤處理器介入
  ↓
1. 記錄詳細錯誤信息（日誌）
2. 判斷錯誤類型
3. 返回統一的錯誤響應
4. 系統繼續運行
```

### 錯誤處理層次

```
1. 函數級錯誤處理（try/except）
   ↓ 如果沒有處理
2. 端點級錯誤處理（裝飾器）
   ↓ 如果沒有處理
3. 全局錯誤處理器（最後防線）
   ↓
4. Flask 默認錯誤處理（最壞情況）
```

- ✅ 所有錯誤返回統一格式

## 全局錯誤處理器 vs 局部錯誤處理

### 局部錯誤處理（try/except）

**優點：**
- 針對特定錯誤進行精確處理
- 可以根據錯誤類型採取不同措施
- 性能更好（不需要全局處理）

**缺點：**
- 需要每個地方都寫 try/except
- 容易遺漏某些錯誤
- 代碼重複

**適用場景：**
- 已知的、可以預期的錯誤
- 需要特殊處理的錯誤
- 性能敏感的場景

### 全局錯誤處理器

**優點：**
- 統一處理所有未捕獲的錯誤
- 不需要每個地方都寫 try/except
- 確保沒有錯誤被遺漏
- 統一錯誤響應格式

**缺點：**
- 無法針對特定錯誤進行精確處理
- 可能掩蓋某些應該特殊處理的錯誤

**適用場景：**
- 未預期的錯誤
- 最後的防線
- 統一錯誤響應

### 最佳實踐

**結合使用：**
```
1. 局部錯誤處理（try/except）
   - 處理已知的、可預期的錯誤
   - 進行精確的錯誤處理

2. 全局錯誤處理器
   - 作為最後防線
   - 處理未預期的錯誤
   - 統一錯誤響應
```

## 實際例子

### 例子1：API 端點

**沒有全局錯誤處理器：**
```python
@app.route('/api/orders/list')
def get_orders_list():
    # 如果這裡出錯，沒有 try/except
    conn = get_db_connection()  # 可能失敗
    cursor = conn.cursor()
    cursor.execute("SELECT * FROM ...")  # 可能失敗
    # 任何錯誤都會導致 500 錯誤
    return jsonify(data)
```

**有全局錯誤處理器：**
```python
@app.route('/api/orders/list')
def get_orders_list():
    # 即使這裡沒有 try/except
    conn = get_db_connection()  # 如果失敗
    # 全局錯誤處理器會捕獲
    # 記錄錯誤、返回友好響應
    return jsonify(data)

@app.errorhandler(Exception)
def handle_exception(e):
    # 統一處理所有錯誤
    logger.error("未預期錯誤", exc_info=True)
    return jsonify({'error': '服務暫時不可用'}), 500
```

### 例子2：不同錯誤類型

**全局錯誤處理器可以處理：**

1. **404 錯誤（資源不存在）**
   ```python
   @app.errorhandler(404)
   def not_found(error):
       return jsonify({'error': '資源不存在'}), 404
   ```

2. **500 錯誤（服務器錯誤）**
   ```python
   @app.errorhandler(500)
   def internal_error(error):
       logger.error("服務器錯誤", exc_info=True)
       return jsonify({'error': '服務器錯誤'}), 500
   ```

3. **所有未處理的異常**
   ```python
   @app.errorhandler(Exception)
   def handle_exception(e):
       logger.error("未預期錯誤", exc_info=True)
       return jsonify({'error': '發生錯誤'}), 500
   ```

## 總結

### 全局錯誤處理器是什麼？

一個**統一捕獲和處理所有未預期錯誤**的機制，作為系統的"最後防線"。

### 為什麼需要？

1. **防止服務崩潰** - 未捕獲的錯誤不會導致整個系統停止
2. **統一錯誤響應** - 所有錯誤返回統一格式
3. **更好的追蹤** - 所有錯誤都記錄在日誌中
4. **安全性** - 不洩漏敏感信息
5. **用戶友好** - 返回友好的錯誤信息

### 如何工作？

1. 捕獲所有未處理的錯誤
2. 記錄詳細錯誤信息（日誌）
3. 返回統一的錯誤響應
4. 系統繼續運行

### 與局部錯誤處理的關係？

- **局部錯誤處理**：處理已知的、可預期的錯誤（精確處理）
- **全局錯誤處理器**：處理未預期的錯誤（最後防線）
- **最佳實踐**：結合使用，互補

**簡單來說：** 全局錯誤處理器就像一個"安全網"，確保即使代碼中沒有明確處理的錯誤，也不會導致系統崩潰，並且會統一處理和記錄所有錯誤。
