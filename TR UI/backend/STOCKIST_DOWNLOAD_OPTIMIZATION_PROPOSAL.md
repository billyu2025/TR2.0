# STOCKIST & TEST REPORT 下載優化方案

## 🔍 問題分析

### 當前實現的問題

1. **同步處理阻塞**
   - 所有文件查找、打包、返回都在同一個 HTTP 請求中完成
   - 請求期間佔用服務器線程/進程，無法處理其他請求
   - 大量文件時，請求時間可能長達數分鐘

2. **內存佔用過高**
   - ZIP 文件在內存中創建
   - 大量文件時，內存佔用可能達到數 GB
   - 可能導致服務器內存不足

3. **文件查找效率低**
   - 如果沒有文件索引，需要遍歷整個文件系統
   - 每個文件都需要檢查是否存在
   - 大量文件時，I/O 操作非常耗時

4. **CPU 密集操作**
   - ZIP 壓縮是 CPU 密集型操作
   - 大量文件壓縮時，CPU 使用率可能達到 100%
   - 影響其他請求的處理

5. **數據庫查詢**
   - 每個 Order 都需要查詢數據庫
   - 沒有批量查詢優化
   - 大量查詢可能導致數據庫鎖定

## 💡 優化方案

### 方案 1：異步任務處理（推薦）⭐

**核心思路：**
- 將下載任務改為異步處理
- 立即返回任務 ID，前端輪詢任務狀態
- 後台線程處理文件查找和打包

**優點：**
- ✅ 不阻塞 HTTP 請求
- ✅ 用戶可以繼續使用系統
- ✅ 可以顯示進度條
- ✅ 支持取消任務
- ✅ 可以並發處理多個任務

**實現方式：**
1. 創建下載任務，返回任務 ID
2. 後台線程處理任務：
   - 查找文件
   - 更新進度（已處理文件數/總文件數）
   - 創建 ZIP 文件
   - 保存到臨時目錄
3. 前端輪詢任務狀態
4. 任務完成後，返回下載鏈接

**技術實現：**
- 使用 `threading.Thread` 或 `concurrent.futures.ThreadPoolExecutor`
- 使用 `download_task_manager.py` 管理任務狀態
- 任務狀態存儲在數據庫中

### 方案 2：流式 ZIP 創建

**核心思路：**
- 使用流式 ZIP 創建，不將所有文件加載到內存
- 邊查找邊打包，邊打包邊發送

**優點：**
- ✅ 內存佔用低
- ✅ 可以立即開始下載
- ✅ 不需要臨時文件

**缺點：**
- ⚠️ 實現複雜
- ⚠️ 無法顯示進度
- ⚠️ 仍然會阻塞請求

**技術實現：**
- 使用 `zipfile.ZipFile` 的流式模式
- 使用 Flask 的 `Response` 生成器
- 使用 `yield` 逐步發送數據

### 方案 3：文件索引優化

**核心思路：**
- 使用文件索引快速查找文件
- 避免文件系統遍歷

**優點：**
- ✅ 查找速度極快（從秒級降到毫秒級）
- ✅ 減少 I/O 操作
- ✅ 降低服務器負載

**實現方式：**
- 使用 `file_index_query.py` 查詢文件
- 預先建立文件索引表
- 定期更新索引

**注意：**
- 系統已經有文件索引功能
- 需要確保索引是最新的

### 方案 4：批量查詢優化

**核心思路：**
- 將多個 Order 的數據庫查詢合併為一次批量查詢
- 減少數據庫往返次數

**優點：**
- ✅ 減少數據庫查詢時間
- ✅ 降低數據庫負載
- ✅ 提高查詢效率

**實現方式：**
```python
# 當前方式（慢）：
for order_no in order_nos:
    query = "SELECT ... WHERE order_no = ?"
    cursor.execute(query, (order_no,))
    result = cursor.fetchone()

# 優化方式（快）：
placeholders = ','.join('?' * len(order_nos))
query = f"SELECT ... WHERE order_no IN ({placeholders})"
cursor.execute(query, order_nos)
results = cursor.fetchall()
```

### 方案 5：ZIP 壓縮優化

**核心思路：**
- 使用更快的壓縮算法
- 或者不壓縮（如果文件已經是 PDF）

**優點：**
- ✅ 減少 CPU 使用
- ✅ 加快打包速度
- ✅ 降低服務器負載

**實現方式：**
```python
# 當前方式（慢，但文件小）：
zipfile.ZipFile(zip_path, 'w', zipfile.ZIP_DEFLATED)

# 優化方式 1（快，但文件大）：
zipfile.ZipFile(zip_path, 'w', zipfile.ZIP_STORED)  # 不壓縮

# 優化方式 2（平衡）：
zipfile.ZipFile(zip_path, 'w', zipfile.ZIP_DEFLATED, compresslevel=1)  # 快速壓縮
```

### 方案 6：分塊處理

**核心思路：**
- 將大量文件分成多個小批次處理
- 每個批次處理一定數量的文件

**優點：**
- ✅ 降低內存峰值
- ✅ 可以顯示進度
- ✅ 降低單次處理負擔

**缺點：**
- ⚠️ 實現複雜
- ⚠️ 總時間可能更長

### 方案 7：緩存機制

**核心思路：**
- 緩存已創建的 ZIP 文件
- 相同請求直接返回緩存文件

**優點：**
- ✅ 重複請求極快
- ✅ 降低服務器負載
- ✅ 提升用戶體驗

**缺點：**
- ⚠️ 需要存儲空間
- ⚠️ 需要緩存失效策略

**實現方式：**
- 使用文件哈希或請求參數生成緩存鍵
- 緩存文件保存 24 小時
- 定期清理過期緩存

## 🎯 推薦方案組合

   - 預期效果：打包時間減少 50%+

### 長期優化（根本解決）

1. **異步任務處理**（方案 1）⭐
   - 改為異步處理
   - 添加進度顯示
   - 預期效果：完全解決阻塞問題

2. **緩存機制**（方案 7）
   - 緩存常用下載
   - 預期效果：重複請求秒級完成

## ⚠️ 注意事項

### 1. 內存管理

- 大量文件時，內存佔用仍然很高
- 考慮使用流式處理或分塊處理
- 監控服務器內存使用情況

- ZIP 文件需要臨時存儲空間
- 異步任務需要存儲任務文件

- 需要定期清理臨時文件

### 3. 並發控制

- 異步任務需要控制並發數量
- 避免同時處理過多任務
- 使用任務隊列管理

- 任務失敗時的處理
- 網絡中斷時的處理
- 超時處理

- 提供錯誤提示

- 提供重試功能
