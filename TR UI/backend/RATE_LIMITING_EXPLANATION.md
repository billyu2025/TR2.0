# 請求限流說明

## 什麼是請求限流？

請求限流（Rate Limiting）是一種**限制用戶或系統在特定時間內發送請求次數**的機制。就像餐廳限制每桌客人的點菜次數，防止有人點太多菜導致廚房忙不過來。

## 類比說明

### 沒有請求限流

**場景：餐廳**
```
顧客A：點100道菜（惡意或誤操作）
  ↓
廚房忙不過來
  ↓
其他顧客等待時間變長
  ↓
整個餐廳服務變慢 ❌
```

### 有請求限流

**場景：餐廳**
```
顧客A：點100道菜
  ↓
服務員：對不起，每桌最多點10道菜
  ↓
顧客A：只能點10道菜
  ↓
廚房正常運作
  ↓
其他顧客正常服務 ✅
```

## 為什麼需要請求限流？

### 問題1：惡意攻擊

**沒有請求限流：**
```
攻擊者：每秒發送1000個請求
  ↓
服務器過載
  ↓
系統崩潰或變慢
  ↓
正常用戶無法使用 ❌
```

**有請求限流：**
```
攻擊者：每秒發送1000個請求
  ↓
限流器：只允許每秒10個請求
  ↓
超過限制的請求被拒絕
  ↓
正常用戶正常使用 ✅
```

### 問題2：資源耗盡

**沒有請求限流：**
```
用戶A：瘋狂刷新頁面（每秒100次）
  ↓
數據庫連接耗盡
  ↓
其他用戶無法查詢
  ↓
系統變慢或崩潰 ❌
```

**有請求限流：**
```
用戶A：瘋狂刷新頁面（每秒100次）
  ↓
限流器：每分鐘最多60次請求
  ↓
超過限制的請求被拒絕
  ↓
資源得到保護
  ↓
其他用戶正常使用 ✅
```

### 問題3：公平分配資源

**沒有請求限流：**
```
用戶A：佔用大量資源（80%）
用戶B：只能使用少量資源（20%）
  ↓
不公平 ❌
```

**有請求限流：**
```
用戶A：最多使用30%資源
用戶B：最多使用30%資源
用戶C：最多使用30%資源
  ↓
公平分配 ✅
```

## 請求限流的工作原理

### 基本概念

**限流規則：**
- **時間窗口**：在多少時間內
- **請求次數**：允許多少次請求
- **超出限制**：超過後如何處理

**示例：**
```
每分鐘最多 10 次請求
每小時最多 100 次請求
每天最多 1000 次請求
```

### 工作流程

```
用戶請求 → 檢查限流規則
  ↓
是否超過限制？
  ├─ 否 → 允許請求，計數+1
  └─ 是 → 拒絕請求，返回錯誤
```

### 限流算法

#### 1. 固定窗口（Fixed Window）

**原理：**
- 將時間分成固定窗口（如每分鐘）
- 每個窗口內允許固定次數的請求
- 窗口重置時計數歸零

**示例：**
```
每分鐘最多 10 次請求

00:00-00:59 → 10次
01:00-01:59 → 10次（重置）
```

**優點：**
- 簡單易懂
- 實現容易

**缺點：**
- 窗口邊界可能出現突發流量
- 不夠平滑

#### 2. 滑動窗口（Sliding Window）

**原理：**
- 使用滑動時間窗口
- 檢查過去一段時間內的請求數
- 更平滑的限流

**示例：**
```
過去1分鐘內最多 10 次請求

當前時間：01:30
檢查：00:30-01:30 的請求數
```

**優點：**
- 更平滑
- 更精確

**缺點：**
- 實現較複雜
- 需要記錄時間戳

#### 3. 令牌桶（Token Bucket）

**原理：**
- 有一個令牌桶，定期添加令牌
- 每次請求消耗一個令牌
- 沒有令牌時拒絕請求

**示例：**
```
桶容量：10個令牌
添加速率：每秒1個令牌

請求1：消耗1個令牌（剩9個）✅
請求2：消耗1個令牌（剩8個）✅
...
請求11：沒有令牌 ❌
```

**優點：**
- 允許突發流量（桶有容量）
- 平滑限流

**缺點：**
- 實現較複雜

## 實際應用場景

### 場景1：API 端點限流

**問題：**
- 訂單列表查詢可能被頻繁調用
- 數據庫壓力大

**解決：**
```
GET /api/orders/list
限流：每分鐘最多 10 次
```

**效果：**
- 防止惡意請求
- 保護數據庫資源
- 公平分配

### 場景2：登入端點限流

**問題：**
- 登入端點可能被暴力破解
- 嘗試大量密碼組合

**解決：**
```
POST /api/auth/login
限流：每分鐘最多 5 次
每小時最多 20 次
```

**效果：**
- 防止暴力破解
- 提高安全性
- 保護用戶賬戶

### 場景3：PDF 生成限流

**問題：**
- PDF 生成消耗資源大
- 可能被大量請求

**解決：**
```
POST /api/pdf/generate
限流：每分鐘最多 3 次
```

**效果：**
- 防止資源耗盡
- 保護服務器
- 確保服務穩定

## 限流策略示例

### 策略1：按 IP 地址限流

**規則：**
```
每個 IP 地址：
- 每分鐘最多 60 次請求
- 每小時最多 1000 次請求
```

**適用：**
- 防止單個 IP 攻擊
- 公平分配資源

### 策略2：按用戶限流

**規則：**
```
每個用戶：
- 每分鐘最多 30 次請求
- 每天最多 10000 次請求
```

**適用：**
- 防止單個用戶濫用
- 保護系統資源

### 策略3：按端點限流

**規則：**
```
訂單列表：每分鐘 10 次
PDF 生成：每分鐘 3 次
登入：每分鐘 5 次
```

**適用：**
- 不同端點不同限制
- 根據資源消耗調整

## 超出限制時的處理

### 方式1：返回錯誤（推薦）

**響應：**
```json
{
    "error": "請求過於頻繁，請稍後再試",
    "code": 429,
    "retry_after": 60
}
```

**HTTP 狀態碼：** 429 Too Many Requests

**優點：**
- 明確告知用戶
- 提供重試時間

### 方式2：排隊等待

**響應：**
```
請求進入隊列
等待處理
```

**優點：**
- 不丟失請求
- 用戶體驗好

**缺點：**
- 可能等待時間長
- 需要隊列管理

### 方式3：降級處理

**響應：**
```
返回簡化版本
或緩存結果
```

**優點：**
- 仍然提供服務
- 用戶體驗好

**缺點：**
- 可能不是最新數據

## 請求限流的好處

### 1. 防止惡意攻擊

- ✅ 防止 DDoS 攻擊
- ✅ 防止暴力破解
- ✅ 防止資源耗盡攻擊

### 2. 保護系統資源

- ✅ 防止數據庫過載
- ✅ 防止 CPU 過載
- ✅ 防止內存耗盡

### 3. 公平分配資源

- ✅ 確保所有用戶都能使用
- ✅ 防止單個用戶佔用過多資源
- ✅ 提高系統穩定性

### 4. 提高系統穩定性

- ✅ 防止系統崩潰
- ✅ 確保服務可用
- ✅ 提升用戶體驗

## 請求限流的缺點

### 1. 可能影響正常用戶

**問題：**
- 正常用戶可能觸發限流
- 需要合理設置限制

**解決：**
- 設置合理的限制
- 提供清晰的錯誤信息
- 允許用戶升級（付費用戶）

### 2. 實現複雜度

**問題：**
- 需要實現限流邏輯
- 需要存儲請求計數

**解決：**
- 使用現成的庫（如 flask-limiter）
- 使用 Redis 存儲計數

### 3. 可能被繞過

**問題：**
- 攻擊者可能使用多個 IP
- 可能使用代理

**解決：**
- 結合其他安全措施
- 監控異常行為
- 使用更複雜的限流策略

## 總結

### 請求限流是什麼？

一種**限制用戶或系統在特定時間內發送請求次數**的機制。

### 為什麼需要？

1. **防止惡意攻擊** - 防止 DDoS、暴力破解
2. **保護系統資源** - 防止資源耗盡
3. **公平分配** - 確保所有用戶都能使用
4. **提高穩定性** - 防止系統崩潰

### 如何工作？

1. 檢查請求是否超過限制
2. 如果超過，拒絕請求或排隊
3. 如果未超過，允許請求並計數

### 實際效果

- ✅ 防止惡意請求
- ✅ 保護系統資源
- ✅ 提高系統穩定性
- ✅ 公平分配資源

**簡單來說：** 請求限流就像餐廳限制每桌點菜數量，防止有人點太多導致廚房忙不過來，確保所有顧客都能正常用餐。
